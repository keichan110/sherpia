# ADR-0007: トリガーを「時間粒度3階層＋毎時コードcron」に再編する

- Status: Accepted
- Date: 2026-06-23

## Context

GASエントリポイント（`src/index.ts`）は現在「機能単位」で定義されている（`processPendingArticles` / `processTrendingQiita` / `processTrendingZenn` / `runGmailDigest` / `runGmailLabelCleanup`）。ここに次の歪み・要求が重なった。

- **claspの命名衝突**: claspは全関数を単一ファイルに展開するため、同名関数はナンバリングされ、GASのトリガー設定画面でどの関数がどのトリガーか判別しづらい。トリガーに紐づける関数は**他から呼ばれないユニークな名前**であってほしい。
- **スケジュールがGAS UI側に散る**: 「何をいつ動かすか」がGASのトリガー設定にしか存在せず、コード上でレビュー・テストできない。トリガー時刻はGAS上ではアバウトにしか管理できない。
- **将来のGmail通知ルーティング要求**: メールのネイティブ通知を完全OFFにし、`action` ラベル付きの「即時通知」をSlackへ、それ以外を「あとで（ダイジェスト）」へ振り分けてフォーカス環境を作りたい。即時レーンには**分オーダーのレイテンシ要件**が生まれる（ただしOTP系はユーザーが能動的に待つためpullで半ば解決し、リアルタイム基盤は今は不要）。
- **6分タイムアウトとコンシューマGmailのクォータ**: 1実行6分、トリガー総実行時間 約90分/日。重い処理（gmail-digest）と軽い処理が混在する。

「全部を1分tickに集約する」案も検討したが、1分tick＝最大1440回/日でクォータの約1/3を「時計を見るだけ」で焼く上、6分制限への集約とcatch-up状態管理を新たに抱える。分オーダー精度を要する処理は現状ゼロ（実カデンスは10分と日次のみ）であり、採用しない。用語の定義は [`CONTEXT.md`](../../CONTEXT.md) を参照。

## Decision

エントリポイントを**時間粒度の階層**で再編し、**毎時以上のスケジュールはコード側の宣言的cronテーブル**で分岐させる。

### 1. GASトリガーは「1時間より細かいカデンス」だけに持たせる

| トリガー関数 | カデンス | 役割 |
|---|---|---|
| `triggerEvery10Minutes` | 10分（GASトリガー） | article-ingest の pending 処理 |
| `triggerHourly` | 毎時（GASトリガー） | 毎時以上のジョブをコード側cronで分岐 |
| `triggerEveryMinute` | 1分（**予約・未作成**） | 将来の即時通知レーン |

- 命名規約: **`trigger` 接頭辞＋カデンス（時刻/頻度）ベース**のユニーク名。`trigger` 接頭辞の関数は**どこからも呼ばれない**（GAS専用エントリポイント）。これでclasp衝突とトリガー選択の迷いを解消する。
- **スロット名はWHEN（いつ）のみを表し、WHAT（何を）は表さない**。`triggerDailyDigest` のように機能をスロット名へ焼き込まない（trends を `dailyMaintenance` に押し込んだ初期案と同じ誤り）。「何をするか」はスケジュールテーブルが語る。
- 毎時より細かいカデンス（10分、将来の1分）だけが専用GASトリガーを持つ。**毎時＝24回/日はクォータ上ほぼ無料**であり、ここでの「時計を見て分岐」は安全（1分tickの毒はクォータに由来する）。

### 2. 毎時以上は宣言的スケジュールテーブル（＝単一の真実）で分岐する

`triggerHourly` は実行時のJST時刻を見て、due なジョブだけを実行する。スケジュールはコード上のテーブルで一元管理する。

```ts
const HOURLY_SCHEDULE = [
  { name: 'gmail-digest:overnight', weight: 'heavy', at: dailyAt(7),    run: () => runGmailDigest() },
  { name: 'trends:qiita',           weight: 'light', at: dailyAt(10),   run: processTrendingQiita },
  { name: 'trends:zenn',            weight: 'light', at: dailyAt(10),   run: processTrendingZenn },
  { name: 'gmail-label-cleanup',    weight: 'light', at: everyHours(3), run: runGmailLabelCleanup },
];
```

- **`dailyAt(h)` は `hour >= h` ＋「当日その枠を実行済みか」ガードで判定する**。これにより7時台の発火がGASにスキップされても、8時台の発火が拾って実行する（**当日catch-up**）。専用日次トリガーには無い自己修復性。
- **`everyHours(n)` は `hour % n === 0` で判定**し、当日ガードは不要（毎回走ってよい）。
- 当日ガードの状態は ScriptProperties に「枠ごとの最終実行日」として持つ。`dailyAt` 系のみ対象。

### 3. このパターンはフラクタルに拡張する

各GASトリガー粒度が、自分の倍数カデンスをコード側テーブルで捌く。将来「20分おき」が来たら `triggerEvery10Minutes` のテーブルに「2回に1回」な行を足すだけで、新しいGASトリガーも新関数も増えない。今は pending しか10分枠に無いため、10分テーブルは「常に実行」1行で済ませ、20分機構は先回りして作らない（構造だけ開けておく）。

### 4. トリガー境界の集約エラーシンク

`runTrigger(name, fn)`（および複数ジョブをまとめる `runCadence`）をトリガー境界に置く。

- 各ジョブを try/catch し、失敗を集めて**全ジョブ実行後に集約rethrow**する（1ジョブの失敗で同枠の他ジョブをスキップしない）。
- catch内で**注入式の `onError` フックを呼ぶ。既定は `log.error`**。GASのトリガー失敗通知メールは「通知しない」に設定し、rethrowは残してGASの実行ダッシュボードを正確に保つ。
- 将来このフックをSlackのエラー専用チャンネルへ差し替える。ただし**Slackエラー通知は本ADRのスコープ外**（現状configはシングルチャンネル前提で、マルチチャンネルconfigスキーマは別途設計する）。

### 5. 重さの宣言と不変条件テスト

- 各ジョブに `weight: 'heavy' | 'light'` を付ける。
- `schedule.test.ts` で0〜23時を総当たりし、各時刻で due になるジョブを集計して**「同一時刻に `heavy` が2つ以上 due になったら fail」**を検証する。`dailyAt`/`everyHours` は時刻に対して決定的なので静的に検出できる。
- これにより「重い枠はhour分散」という設計規約を、コメントではなく `pnpm test` で落ちる不変条件として強制する。6分タイムアウトは本番でしか顕在化しない事故なので、デプロイ前に潰す。
- ジョブの**実測時間**を計測して予算超過を警告する版は将来（`testRun` 系の計測ハーネス）。本ADRは「宣言した重さ」の静的チェックまで。

## Consequences

- GASトリガーは3種（`triggerEvery10Minutes` / `triggerHourly` /（将来）`triggerEveryMinute`）に集約され、設定画面での選択の迷いとclasp衝突が消える。
- 「何をいつ動かすか」がコード上の宣言的テーブルに一元化され、レビュー・テスト可能になる。同一Pipelineを複数スロットへ配線できる（朝・夜の2回ダイジェスト等は `dailyAt(7)` と `dailyAt(20)` の2行で表現し、構造を変えずに拡張できる）。
- `dailyAt` のcatch-upにより、GASのトリガースキップに対して日次ジョブが頑健になる。
- 代償として、**当日実行済みガードの状態（ScriptProperties）を自分で持つ責任**が発生する。専用日次トリガーなら不要だった。ガードが対象なのは `dailyAt` 系のみ。
- ダッシュボードは `triggerHourly` が24回/日（多くは空振り）並ぶため、観測はログと集約エラーシンク側へ寄せる。
- **トリガーの再配線が必要**: 関数名が変わり日次の発火がコード側へ移るため、GASコンソールで旧トリガー削除→新トリガー登録を手動で行う移行手順が発生する。
- 本ADRは構造の確定。即時通知Pipeline（gmail-notify）本体・Slackエラー通知のマルチチャンネルconfig・実測ベースの予算検知は、いずれも本ADRが用意したスロット／フック／テーブルに後から差し込む将来作業とする。
- なお trends の実運用時刻は10時台（CLAUDE.md の「7〜8時」は古い記述）。本ADRのテーブルは実態（`dailyAt(10)`）に合わせる。
