# ADR-0003: gmail-digest の出力をBlock Kit化し親メッセージ＋スレッド返信に分割する

- Status: Accepted
- Date: 2026-06-21

## Context

ADR-0002 で gmail-digest の出力を「**1通のSlackメッセージに集約**」と決めた（2-1: 各メールの件名・送信者・本文冒頭200字を blocks のセクションで並べる）。

実運用イメージを詰めた結果、次の課題が見えた。

- 日次で届くNewsletterを1通に詰め込むと、件数が多い日にチャンネルが長文で埋まり、**一覧性が落ちて流れてしまう**。
- Block Kit はメッセージあたり**50ブロック上限**があり、件数が多い日は抵触しうる。

そこで、チャンネルに流す情報を最小化し、詳細はスレッドに畳む構成へ見直す。用語の定義は [`CONTEXT.md`](../../CONTEXT.md) を参照。

## Decision

### 出力構造を「親メッセージ＋スレッド返信」に変更

- **親メッセージ**: `📬 昨日のNewsletter` ＋ 日付・件数のみ。チャンネルに流れるのはこれだけにし、一覧性を保つ。
- **詳細**: 親メッセージの `ts` を `thread_ts` に指定した**スレッド返信**として投稿する。
- **0件のときは親メッセージのみ**（スレッド返信なし）。親で「0件」を伝え、ジョブ生存確認は維持する。

### 詳細はBlock Kitカードで表現

- 1メール = **section**（件名太字＋本文冒頭200字＋「メールを開く」ボタン）＋ **context**（送信者）＋ **divider**。
- 「メールを開く」は Block Kit の **url ボタン**。クリックで直接Gmailを開き、interactionハンドラ（サーバ側応答）は不要。Gmailを開くURLは `GmailThread.getPermalink()` を使う。
- 送信者は context ブロックに `*表示名* &lt;email&gt;` で表示する。context は小さめ・グレー表示になりGmail風の控えめな見た目になる。表示名が無い送信者はアドレスのみにフォールバックする。`<>` は mrkdwn のリンク記法と衝突するため `&lt;` / `&gt;` でエスケープする。
- **アイコンは出さない**。正確なGmailアバターはAPIで取得できず、近似アイコン（ファビコン等）は不正確で混乱を招くため採用しない。

### 10件単位でチャンク分割

- スレッド返信は**最大10件/通**。30件なら3返信に分割する。
- 理由: Block Kit の50ブロック上限の回避（1メール≒3ブロック）と、スレッド内の可読性確保。

### Capability への影響（薄さは維持）

- `slack.postMessage` を拡張: `blocks` を受け取れるようにし、`thread_ts` を指定でき、戻り値で `ts` を返す（親→スレッド紐付け）。`text` は通知フォールバックとして残す。**整形・分割・チャンク化の判断はPipelineが所有**する。
- `gmail` に `getThreadPermalink(thread)` 相当の薄いラッパを追加（`thread.getPermalink()`）。送信者文字列（`"表示名 <email>"`）のパースはPipeline側のドメインロジックとして持つ。

## Consequences

- ADR-0002 の「出力は1通のSlackメッセージに集約」は本ADRで**更新**される。ダイジェスト型でも Sink 呼び出しは**複数回**（親＋スレッド返信N通）になりうる。CONTEXT.md のダイジェスト型定義に注記を追加した。
- チャンネルの一覧性が上がり、詳細はスレッドに隔離される。
- 2-2（Geminiによる横断要約）を導入する際は、親メッセージまたはスレッド先頭に「まとめ」段落を置く形へ再整理する（本ADRの構造の上に載せる）。
- url ボタンのみのため Slack App の interactivity 設定は不要だが、App設定によっては未構成警告が出る可能性があり、実機で確認する。
- `getThreadPermalink()` のURLが実機で意図どおりメールを開くか、デプロイ後に確認する。
