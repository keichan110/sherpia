# ADR-0005: gmail-digest フェーズ2-2 の要約を「ページ単位の各メール要約」に変更する

- Status: Accepted
- Date: 2026-06-21

## Context

ADR-0002 のフェーズ2-2 は当初「前日ウィンドウのメール群を **Gemini で1回だけ横断要約**し、ダイジェスト先頭に『まとめ』段落（⚠️対応が必要／📊種類別／📝全体要約）を置く」設計で実装した（直前のコミット）。

運用イメージを詰める中で、この設計には次の不一致が見えた。

- **本当に欲しいのは「各メールの中身の要約」**だった。「このメールは要約すると何か」を1通ずつ掴みたいのに、横断要約1回では個々のメールの中身が埋もれる。フェーズ1の「本文200字切り捨て」も中身の理解には足りない。
- メールマガジンが対象なので、**「対応が必要」な項目は基本的に存在しない**。セミナー申込期限・イベント開催日などの**日付さえ取りこぼさなければ十分**で、`actionItems`／`categories` は過剰だった。
- **全体要約はそれだけで Gemini リクエストを1消費する**。各メール要約があれば全体像は把握でき、全体要約の費用対効果が低い。
- 全件を1リクエストに詰めて個別要約させると、**件数が増えるほど精度が落ちる**（lost in the middle、中盤メールの日付の取りこぼし）。

GAS のタイムトリガー実行には時間的余裕があり、Gemini を複数回呼ぶ余地はある。一方で Gemini API の **RPM は 15** で、1実行あたりのリクエスト数は確実にこの範囲に収める必要がある。

## Decision

### 各メールを「ページ単位」で要約する（横断・全体要約は廃止）

- Gemini は**各メールの本文要約だけ**に使う。出力は1通あたり `{ subject, summary }`。`overview`／`actionItems`／`categories` は廃止する。
- 全件まとめてではなく **ページ単位（既定5件）で1リクエスト**にまとめて要約する。1リクエストの同時投入件数を抑え、中盤の劣化・日付の取りこぼしを防ぐ。
- 要約は1〜2文で簡潔に。ただし**セミナー申込期限・締切・イベント開催日などの日付は省略しない**ことをプロンプトで明示する。
- **ページサイズは単一の定数**で管理し、「1リクエストで要約に同時投入する件数」と「1スレッド返信あたりの件数」を**同じ値で兼ねる**（要約精度と RPM のトレードオフを1か所で調整できる）。

### Gemini 呼び出し回数と RPM 予算

- 1実行あたりのリクエスト数は `ceil(件数 / ページサイズ)`。全体要約を廃止したため、以前案の `+1` は不要。
- RPM 15 と 503 リトライ（`callGeminiAPI` は最大3回）の余裕を見て、**Gemini 経路の上限を 70 件**（既定ページサイズ5で最大14リクエスト）とする。
- ページ要約とその Slack 返信を**交互に**行うため、Gemini 呼び出しが時間的に分散し RPM 圧も下がる。

### 上限超過時はフェーズ1方式へフォールバック

- 件数が上限を超えたら **Gemini を一切呼ばず**、フォールバックとして**本文を200字に切り捨てた従来方式**で返信する（`truncateBody` を復活）。親メッセージに「件数が多いため要約は省略し本文冒頭のみ表示」する旨を注記する。
- 本文のトリムは**フォールバック時のみ**。Gemini 入力では本文をトリムしない（どこに書かれるか分からない日付を切り落とさないため）。

### Slack 出力

- 親メッセージは **header ＋ 件数のみ**。
- スレッド返信（ページ単位）の各メール欄に **件名・送信者・要約・「メールを開く」リンク**を並べる。**末尾の全体要約は持たない**。
- 0件のときは従来どおり Gemini を呼ばず「0件」だけ通知する。

## Consequences

- フェーズ2-2 の Gemini 呼び出しは「ウィンドウあたり1回」から「**ページあたり1回（`ceil(件数/ページサイズ)`）**」に変わった。CONTEXT のダイジェスト型の例示「30件のメール → Gemini 1回」は、gmail-digest では「ページ単位で複数回」に当たる（ダイジェスト型の原則『まとめて加工しSinkはまとめてN通』自体は維持）。
- ADR-0002 のフェーズ2-2 で述べた `actionItems`／`categories`／全体要約・親メッセージへの『まとめ』段落は本 ADR で取り消し、各メール要約に置き換える。
- 出力レイアウトは「親＝件数のみ／返信＝各メールの要約一覧」に確定。フェーズ1の本文冒頭はフォールバック経路にのみ残る。
- 要約精度はページサイズに依存する。既定5から始め、実際の要約品質を見て定数1か所で調整する。RPM 上限（15）とページサイズ・件数上限（70件）の関係は本 ADR の前提であり、ページサイズを変える際は件数上限も併せて見直す。
- メール件数が恒常的に上限を超える場合は要約が一切出なくなる（フォールバックのみ）。実需が出た時点で上限・ページサイズの調整、または分割実行を別途検討する。
