## 目的

PRの差分から主な変更内容を日本語の箇条書きでまとめ、`gh pr edit` でPR本文に直接書き込む。後から GitHub 上で振り返れる変更記録として残す。

## 手順

### ステップ1: PRを特定する

**ユーザーがPR URLまたはPR番号を提示している場合:**
```bash
gh pr view <URL_or_number> --json number,title,url,body,baseRefName,headRefName
```

**現在のブランチから自動取得する場合:**
```bash
gh pr view --json number,title,url,body,baseRefName,headRefName
```
PRが見つからない場合は、ユーザーにPR URLまたはPR番号を尋ねる。

### ステップ2: 変更内容を取得する

変更ファイルの一覧を把握する:
```bash
gh pr view <number> --json files --jq '.files[] | "\(.status)\t\(.path)"'
```

差分の全体を取得する:
```bash
gh pr diff <number>
```

差分が大きい場合（500行超）はコミットメッセージも参照する:
```bash
gh pr view <number> --json commits --jq '.commits[].messageHeadline'
```

### ステップ3: 変更内容を分析してまとめる

差分とファイル一覧から以下の観点で変更を整理する:

- **機能追加**: 新しく追加された機能・処理・エンドポイントなど
- **機能変更・修正**: 既存の動作を変えた箇所
- **リファクタリング**: 動作は変えずにコードを整理した変更
- **設定・依存関係**: package.json, 設定ファイル, 環境変数などの変更
- **削除**: 不要になったコードや機能の削除
- **その他**: ドキュメント, テスト, CI設定など

**まとめ方のルール:**
- インデント修正・タイポ修正・コメントの微調整など些細な変更は省略する
- 同じカテゴリの小さな変更は1行にまとめる（例:「〇〇関連の定数を整理」）
- 何を変えたかを中心に、必要に応じて理由も1語で補足する（例:「〇〇をXXに変更（パフォーマンス改善）」）
- 箇条書きは3〜7項目を目安とする（PRの規模に応じて調整）
- 日本語で記述する。技術的な固有名詞（ライブラリ名・関数名など）はそのまま使ってよい

### ステップ4: PRタイトルを提案する

現在のタイトルと変更内容をもとに、適切なタイトルを提案する。

**タイトルのルール:**
- 50文字以内で変更の本質を端的に表す
- 日本語で記述する（技術的な固有名詞はそのまま）
- 命令形や動詞で始める（例:「〇〇機能を追加」「〇〇をXXに変更」「〇〇のバグを修正」）
- gitmoji は使わない

現在のタイトルが既に適切な場合はそのまま維持する。提案タイトルをユーザーに提示し、承認を得てから更新する。

### ステップ5: PR本文に書き込む

以下の本文テンプレートを生成する:

```
## 変更内容

- <変更1>
- <変更2>
- <変更3>
```

**既存のbodyがある場合の扱い:**
- 既存のbodyが空またはホワイトスペースのみ → そのまま上書き
- 既存のbodyに内容がある → ユーザーに確認する（「上書きしますか？それとも追記しますか？」）

`gh pr edit` でPRタイトルと本文を同時に更新する:
```bash
gh pr edit <number> --title "<新しいタイトル>" --body $'## 変更内容\n\n- <変更1>\n- <変更2>'
```

改行を含む本文は heredoc または `$'...'` 記法で渡す。

更新後、ユーザーにPRのURLを伝えて完了を報告する。
