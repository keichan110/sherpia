## 目的

PRの差分を分析し、リポジトリの標準テンプレート（`.github/PULL_REQUEST_TEMPLATE.md`）に沿った日本語のdescriptionを作成して、`gh pr edit` でPR本文に書き込む。レビュアーが「何を・なぜ変えたか」を素早く把握でき、後から GitHub 上で振り返れる変更記録として残すことを狙う。

## 手順

### ステップ1: PRを特定する

**ユーザーがPR URLまたはPR番号を提示している場合:**
```bash
gh pr view <URL_or_number> --json number,title,url,body,baseRefName,headRefName
```

**現在のブランチから自動取得する場合:**
```bash
gh pr view --json number,title,url,body,baseRefName,headRefName
```
PRが見つからない場合は、ユーザーにPR URLまたはPR番号を尋ねる。

### ステップ2: 変更内容を取得する

変更ファイルの一覧を把握する:
```bash
gh pr view <number> --json files --jq '.files[] | "\(.status)\t\(.path)"'
```

差分の全体を取得する:
```bash
gh pr diff <number>
```

差分が大きい場合（500行超）はコミットメッセージも参照する:
```bash
gh pr view <number> --json commits --jq '.commits[].messageHeadline'
```

### ステップ3: テンプレートを読み込む

descriptionの土台はリポジトリ標準テンプレート。最初に読み込み、見出し構成をそのまま骨組みとして使う:
```bash
cat .github/PULL_REQUEST_TEMPLATE.md
```
テンプレートが存在しない場合は、後述の標準構成（概要 / 変更内容 / 補足）で代用する。テンプレート内の `<!-- ... -->` コメントは記入ガイドなので、生成する本文には残さない。

### ステップ4: 各セクションを埋める

差分とファイル一覧をもとに、テンプレートの各セクションを日本語で埋める。テンプレートはあくまで骨組みで、中身はPRの実態に合わせて書く。

**## 概要**
- このPRの目的・背景を1〜3文で。「何を」「なぜ」やるPRなのかが伝わるように書く。
- 定義・決定の詳細はリポジトリのドキュメント（`CONTEXT.md`・`docs/adr/` など）を参照し、本文では重複させない。「詳細は ADR-0001 を参照」のように導線を示すと良い。

**## 変更内容**
- ファイル/領域単位で「何を変えたか」を箇条書きにする。新規追加・変更・削除が読み取れるようにする（例:「`src/foo.ts`（新規）— 〇〇を実装」）。
- 何を変えたかを中心に、必要に応じて理由を1語添える（例:「〇〇をXXに変更（パフォーマンス改善）」）。
- インデント修正・タイポ修正・コメントの微調整など些細な変更は省略する。同じカテゴリの小さな変更は1行にまとめる。
- 箇条書きは3〜7項目を目安に、PRの規模に応じて調整する。

**## 補足**（任意）
- レビュー時の注意点・残課題・スコープ外にしたこと・関連PRやIssueへの参照などを書く。
- 書くことが無ければセクションごと省略してよい。空の「補足」見出しだけを残さない。

**PR固有のセクション（任意）**
- そのPRでレビュアーに特に伝えたい観点があれば、見出しを追加してよい。たとえば規約やルールを定めるPRなら「## 規約の要点」、互換性に影響するなら「## 移行手順」など。
- これらは固定の項目ではなく、PRの内容が要求するときだけ足す。無理に埋めない。

技術的な固有名詞（ライブラリ名・関数名・ファイルパスなど）はそのまま使ってよい。

### ステップ5: PRタイトルを提案する

現在のタイトルと変更内容をもとに、適切なタイトルを提案する。

**タイトルのルール:**
- 50文字以内で変更の本質を端的に表す。
- 日本語で記述する（技術的な固有名詞はそのまま）。
- このリポジトリの慣習に合わせ、「〜する／〜した」の終止形で締める（例:「コーディング規約を追加する」「git管理対象から除外する」「デフォルトモデルを変更しAPI呼び出しを改善する」）。
- gitmoji は使わない。

現在のタイトルが既に適切な場合はそのまま維持する。提案タイトルをユーザーに提示し、承認を得てから更新する。

### ステップ6: PR本文に書き込む

ステップ4で埋めた内容を、テンプレートの見出し構成のまま本文として組み立てる。記入ガイドのコメントは残さない。生成例:

```
## 概要

<このPRの目的・背景>

## 変更内容

- <変更1>
- <変更2>
- <変更3>

## 補足

<注意点・残課題など。無ければこのセクションごと省く>
```

**既存のbodyの扱い:**
- 既存のbodyが空またはホワイトスペースのみ → そのまま上書き。
- 既存のbodyに内容がある → ユーザーに確認する（「上書きしますか？それとも追記しますか？」）。

`gh pr edit` でPRタイトルと本文を同時に更新する:
```bash
gh pr edit <number> --title "<新しいタイトル>" --body "$(cat <<'EOF'
## 概要

<このPRの目的・背景>

## 変更内容

- <変更1>
- <変更2>

## 補足

<注意点・残課題など>
EOF
)"
```

改行を含む本文は heredoc または `$'...'` 記法で渡す。

更新後、ユーザーにPRのURLを伝えて完了を報告する。
